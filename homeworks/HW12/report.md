# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использовался датасет S12-hw-dataset.csv, содержащий временной ряд с колонками date и target. Прогноризовалось следующее значения ряда
- Как был организован `train / validation / test` по времени: в соотношении 70/15/15 с сохранением хронологического порядка
- Какие подходы сравнивались (`B1`, `B2`, `B3`, `R1`):
1. B1 - наивный прогноз (последнее значение)
2. B2 - скользящее среднее,
3. B3 - линейная модель Ridge на лаговых, rolling и календарных признаках
4. R1 - рекуррентная нейросеть GRU
- Лучший результат показала модель R1 (GRU), так как она смогла учитывать временную динамику ряда и длинные временные зависимоти

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: target
- Частота временного ряда: часовая
- Горизонт прогноза: 1 шаг вперед
- Размер датасета: 4320 наблюдений
- Диапазон дат: 2025-01-01 00:00:00 — 2025-06-29 23:00:00
- Есть ли дополнительные внешние признаки: нет
- Временной ряд показывает колебание значений, присутсвует ежедневная сезонность: в дневные часы значения растут, а в ночные падают. При этом ропусков нет. Со временем среднее значение немного растет, что указывает на рост тренда. Такой набор данных хорошо подходит прогнозирования с помощью моделей, учитывающих временной контекст

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: в соотношении 70/15/15
- Какие доли или интервалы использовались: разбиение выполнено по времени, без перемешивания
- Почему `random split` здесь некорректен: потому что при случайном разбиении модель может видеть информацию из будущего, которая в реальной задачи не может быть известна

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: day_of_week
- Как обрабатывались пропуски после построения лагов/rolling: удалялись (dropna)
- Использовалось ли масштабирование и где именно оно обучалось: не использовалось для baseline моделей (только для GRU - StandardScaler на train)

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): прогноз равен последнему известному значению ряда
- B2 (`moving-average`): прогноз на основе скользящего среднего с окном 7
- B3 (`ridge-lag-features`): линейная регрессия Ridge с лаговыми, rolling и календарными признаками.
- R1 (`gru-forecast`): GRU-модель, обученная на последовательностях длиной 28

Дополнительно:

- Основная метрика выбора лучшей модели: RMSE
- Дополнительные метрики: MAE, MAPE
- `window_size` для `GRU`: 28
- Batch size: 64
- Максимальное число эпох: 30
- Optimizer: Adam
- Learning rate: 1e-3
- Как сохранялась лучшая модель: на основе лучшего val_MSE

## 7. Анализ

(8-15 предложений)

Наивный baseline показал хороший результат, что говорит о сильной автокорреляции временного ряда: значение в момент времени сильно зависит от предыдущего. `moving average` оказался хуже, а использование лаговых и rolling-признаков никак не улучшило результат. GRU показала лучший результат, поскольку умеет учитывать последовательную структуру данных и зависимости на длинном горизонте. В ряду заметна сезонность, и GRU лучше ее улавливает, чем простые модели. Потенциальные утечки данных могли возникнуть при использовании random split, так как в таком случае модель видела бы признаки из будущего. Выбор модели по validation принципиален, так как test используется только для финальной проверки и не должен влиять на обучение. Ошибки модели наиболее заметны в точках резких изменений ряда

## 8. Итоговый вывод

(3-7 предложений)

Для данной задачи базовым подходом можно считать модель GRU, так как она показывает наилучшее качество благодаря учету временной структуры данных. Простые baseline-модели, особенно naive-last, могут быть достаточно эффективны при высокой взаимосвязи ряда, когда следующие признаки сильно зависят от предыдущих. Переход к GRU или другим нейросетям оправдан при наличии сложной динамики и сезонности. Главный вывод - для временных рядов критически важно использовать корректное разбиение данных, и контролировать их таким образом, чтобы модель не знала данные их будущего, а выбор модели при этом зависит от сложности данных
